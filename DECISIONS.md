## 2026-07-07 — Прод-сервер зачекаучен на `main` (а не `feature/deploy`)

- **Ствол проекта — ветка `main`; сервер обязан стоять на ней.**
  Обнаружили, что прод с этапа 6 сидел на `feature/deploy` (застывшая ветка).
  Автодеплой делает `git pull` ТЕКУЩЕЙ ветки сервера — значит хостовые файлы
  (compose, Caddyfile, `backup.sh`) на сервер не приезжали, хотя образ `app`
  обновлялся (он идёт из GHCR отдельным каналом). Симптом: смердженный в `main`
  скрипт бэкапа не появился на сервере. Починка: `git checkout main` на сервере
  + перевод стека на образ из GHCR (`pull app` + `up -d`). На будущее — правило
  в `docs/deploy.md`: сервер всегда на `main`.
- **Два независимых канала доставки на прод.** Образ `app` едет через GHCR
  (`docker compose pull app`, не зависит от git-ветки), а конфиги/скрипты — через
  `git pull`. Полезно помнить при отладке: «приложение свежее, а конфиг старый»
  = проблема во втором канале (ветка/pull), не в первом.

## 2026-07-07 — Этап 7.1: бэкап БД

- **Только локальный бэкап, без офсайт-копии в Object Storage.**
  Hetzner Object Storage платный (от ~€4.90/мес за бакет), для пет-проекта на
  этом этапе не оправдано. `docker/backup.sh` кладёт `pg_dump` в `~/backups`
  на диске хоста — вне docker volume `pgdata`, чтобы копии не пропали вместе
  с ним при случайной пересборке/удалении тома. Это защищает от плохой
  миграции/случайного `DELETE`, но не от потери сервера целиком. Следующий
  шаг при росте проекта — rclone + бесплатный S3-совместимый бакет
  (Backblaze B2 / Cloudflare R2).
- **Хранение 14 дней, не 7.** Решили с запасом на случай, если проблему
  заметят не сразу.
- **`pg_dump` внутри контейнера `db` через `docker compose exec`, а не отдельным
  клиентом на хосте.** Не нужно ставить `postgresql-client` на хост и
  синхронизировать версию с версией сервера в контейнере — используется тот
  же бинарник, что и сама БД. Плюс `POSTGRES_USER`/`POSTGRES_DB` уже есть в
  окружении контейнера, скрипту не надо самому парсить `.env`.

## 2026-07-07 — Этап 7: CI/CD (GitHub Actions)

- **Два workflow-файла, а не один: `ci.yml` (гейт) и `deploy.yml` (конвейер).**
  У них разные триггеры и разные гарантии. `ci.yml` на `pull_request` — дешёвая
  отменяемая проверка перед merge (линт + тесты). `deploy.yml` на `push` в main —
  повторяет тесты и только потом собирает/деплоит (`needs:`), поэтому сломанный
  main не доедет до сервера даже при пуше мимо PR. Тест-джоба продублирована
  между файлами (~25 строк) осознанно: на этом масштабе дублирование дешевле,
  чем reusable workflow (`workflow_call`) — это известный рефакторинг на потом.

- **Тесты в CI — на реальном Postgres через service container.**
  Конфиг service-контейнера (user/pass/db `app/app/link_shortener_test`,
  порт `5433:5432`, healthcheck `pg_isready`) в точности повторяет
  `docker-compose.test.yml` и жёстко зашитый URL в `tests/conftest.py`.
  Тестируем тем же движком, что в проде (не SQLite).

- **Образ в GHCR + `GITHUB_TOKEN`, пакет публичный.**
  Пуш в ghcr.io автоматическим токеном (`permissions: packages: write`) —
  без личных PAT: токен выпускается на один запуск и умирает с job'ом. Пакет
  сделан публичным → серверу для `docker pull` не нужны никакие креды registry:
  на проде не хранится ни одного токена, который надо ротировать или который
  утечёт. Оправдано тем, что репозиторий и так открыт (портфолио). Два тега:
  `latest` (его тянет прод) и `<SHA>` (неизменяемый снимок → аудит и откат).

- **`build once, run everywhere`: прод берёт готовый образ, не собирает.**
  В `docker-compose.prod.yml` у app теперь `image: ghcr.io/...:latest` +
  `build: !reset null` (тег `!reset` убирает `build: .` из базового файла при
  слиянии). На сервер едет тот самый артефакт, что прошёл тесты и сборку в CI,
  а не пересборка из кода на слабой CX22. Базовый `docker-compose.yml` сохраняет
  `build: .` для локального dev.

- **Автодеплой — SSH из Actions обычным `ssh`, а не сторонним action.**
  Отдельная CI-пара ключей (не личная): скомпрометируют — удаляем одну строку
  в `authorized_keys`, личный доступ цел (наименьшие привилегии). Host key
  сервера пиннится через секрет `SSH_KNOWN_HOSTS` вместо `StrictHostKeyChecking=no`
  (защита от MITM). Plain `ssh` вместо appleboy/ssh-action: не отдаём приватный
  ключ стороннему коду и видно реальную механику. На сервере `set -e` +
  `git pull --ff-only` (падает, если файлы правили руками) + `pull app` + `up -d`.

- **Защита ветки main через ruleset.**
  Require PR + обязательные чеки `lint` и `test` перед merge, запрет force-push
  и удаления. Тест-джоба в `deploy.yml` всё равно нужна: страхует merge-race —
  два по отдельности зелёных PR могут вместе сломать main.

- **Известное упрощение (на будущее).** Деплой по подвижному тегу `latest` —
  ок для пет-проекта. Прод-грейд: деплоить по SHA-тегу (полностью
  детерминированный откат), digest-пиннинг, staging-окружение перед продом.

## 2026-07-06 — Этап 6: Деплой на Hetzner

- **Caddy как reverse proxy, а не nginx + certbot.**
  Caddy получает и продлевает сертификаты Let's Encrypt автоматически (ACME HTTP-01),
  редиректит http→https сам — весь конфиг 3 строки. Связка nginx+certbot требует
  ручного конфига, отдельного certbot и cron на продление. nginx ещё появится
  в проекте на этапе 8 (раздача статики фронтенда).

- **`docker-compose.prod.yml` — override, а не отдельный полный файл.**
  Запуск: `docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d`.
  Структура стека описана один раз в базовом файле, prod-файл содержит только
  отличия: сервис caddy, обязательные секреты (`${VAR:?}` вместо `${VAR:-default}`),
  закрытые порты app/db (`ports: !override []` — тег `!override` заменяет список
  целиком вместо слияния; нужен compose 2.24+).

- **uvicorn `--proxy-headers` + `FORWARDED_ALLOW_IPS` для работы за прокси.**
  За Caddy TCP-соединение приходит от прокси, и `scope["client"]` — IP Caddy;
  rate-limiter считал бы всех пользователей одним клиентом. С флагом uvicorn берёт
  реальный IP из `X-Forwarded-For`. Заголовку верим только от доверенных адресов:
  дефолт 127.0.0.1 (dev как раньше), на проде `*` — безопасно, т.к. порт 8000
  наружу не публикуется и достучаться до app может только Caddy.

- **Firewall на уровне Hetzner Cloud (22/80/443), а не ufw на хосте.**
  Docker вписывает свои правила iptables впереди ufw — опубликованный порт
  контейнера обходит ufw. Облачный firewall фильтрует до того, как пакет
  дошёл до машины, и от Docker не зависит.

- **Hardening-минимум: пользователь deploy + sudo, вход только по SSH-ключу.**
  `PasswordAuthentication no` (в т.ч. в `sshd_config.d/50-cloud-init.conf` —
  cloud-init переопределяет главный конфиг!), unattended-upgrades для
  security-обновлений. Боты сканируют новый сервер с первых минут — в логах app
  видны переборы `/.env`, `/.git/HEAD`, ключей облаков; всё отбито 404/429.

- **Сборка образа на сервере (`up --build`), деплой руками — осознанно до этапа 7.**
  Без registry деплой = `git pull && docker compose up -d --build`. На этапе 7
  (CI/CD) заменится на pull готового образа из GHCR.

- **Прод-секреты генерируются на сервере и не покидают его.**
  `.env` на сервере создан вручную (`secrets.token_hex/token_urlsafe`), в git
  не попадает, на ПК разработчика не хранится.

- **Известное ограничение: IPv6-клиенты видны как 172.18.0.1.**
  Docker без включённого IPv6 прогоняет такие соединения через userland-proxy,
  теряя исходный адрес; rate-limiter считает всех IPv6-клиентов одним. Лечится
  включением IPv6 в Docker-демоне — отложено как некритичное.

## 2026-07-06 — Этап 5: Docker

- **Multi-stage build (builder → runtime).**
  В стадии `builder` живёт uv и ставятся зависимости в `.venv`; в финальный образ
  `runtime` копируется только `.venv` и код. uv, кэши и dev-инструменты не попадают
  в прод-образ → меньше размер (~68 МБ сжатый) и меньше поверхность атаки.

- **Порядок слоёв: сначала `pyproject.toml` + `uv.lock`, потом код.**
  Слой установки зависимостей зависит только от манифестов, поэтому правки в `app/`
  не пересобирают все зависимости заново — работает кэш Docker. Плюс
  `--mount=type=cache` для кэша загрузок uv между сборками.

- **Non-root пользователь `app` (uid 999) в контейнере.**
  По умолчанию процесс идёт от root; при уязвимости это шаг к побегу на хост.
  Обычный пользователь ограничивает ущерб (стандарт security-ревью).

- **`python:3.13-slim`, не Alpine.**
  У Alpine musl вместо glibc — готовые wheel (asyncpg, bcrypt) могут не встать
  или собираться из исходников. slim = Debian без лишнего, но с glibc.

- **Миграции в entrypoint контейнера (`alembic upgrade head` перед uvicorn).**
  Для одного инстанса просто и надёжно. `exec` передаёт uvicorn PID 1 → корректный
  graceful shutdown по SIGTERM. Альтернатива (init-контейнер/job) понадобится
  на этапе 7 (K8s), когда реплик несколько и параллельный накат недопустим.

- **Секреты через подстановку `${VAR}` из `.env`, не в compose-файле.**
  `docker-compose.yml` коммитится в git, поэтому секреты (SECRET_KEY, пароль БД)
  тянутся из `.env` (в git не попадает). Дефолты `:-` оставлены только для учебного
  запуска «из коробки»; в проде их убирают, чтобы стек падал без заданного секрета.

- **Named volume `pgdata` для БД, в отличие от `tmpfs` у тестовой БД.**
  Данные dev-БД должны переживать `docker compose down`; тестовой — наоборот,
  стираться (нужна чистота между прогонами).

- **`db_echo` вынесен в настройки (по умолчанию False).**
  Раньше `echo=True` был захардкожен → в проде весь SQL сыпался в логи
  (шум + риск засветить данные). Теперь включается через `DB_ECHO=true` в `.env`.

- **Compose-сервис `db` заменил ручной `docker run postgres` из CLAUDE.md.**
  Один источник правды для dev-БД; гибридный режим (БД в Docker, app локально
  через uv) продолжает работать через порт 5434.

## 2026-07-04 — Этап 4: Тестирование

- **Реальный PostgreSQL в тестах, не SQLite in-memory.**
  SQLite отличается от Postgres по типам, ограничениям и поведению транзакций.
  Отдельный контейнер на порту 5433 с `tmpfs` (данные в RAM → быстро и одноразово).

- **Изоляция тестов через SAVEPOINT + rollback, а не пересоздание таблиц.**
  Таблицы создаются один раз за сессию (`create_all`). Каждый тест работает внутри
  транзакции (`conn.begin()`), а `session.commit()` в сервисах делает `RELEASE SAVEPOINT`,
  не настоящий COMMIT. После теста — `trans.rollback()` → данные исчезают.
  Это на порядок быстрее, чем `drop_all` + `create_all` для каждого теста.

- **`asyncio_default_test_loop_scope = "session"` — один event loop на все тесты.**
  Без этого session-scoped фикстура `engine` создаёт подключения в одном loop,
  а function-scoped тесты работают в другом → `RuntimeError: Future attached to a different loop`.

- **Сброс rate limiter между тестами.**
  Middleware хранит состояние в памяти. Все тесты идут от одного IP (127.0.0.1),
  поэтому без сброса после 30 запросов остальные получают 429.
  Решение: в фикстуре `client` находим `RateLimitMiddleware` в стеке и чистим `requests`.

- **pytest + pytest-asyncio + httpx — стек для async FastAPI-тестов.**
  `httpx.AsyncClient` с `ASGITransport` — нативный async HTTP-клиент.
  `TestClient` из Starlette — sync, не подходит для async зависимостей без хаков.

- **75 тестов, 92% покрытия.**
  Unit: schemas, services (auth/user/link), middleware.
  Integration: все HTTP-эндпоинты (auth, links, redirect), dependency `get_current_user`.

## 2026-07-04 — Этап 3: Авторизация и middleware

- **Alembic для миграций вместо `create_all`.**
  `create_all` умеет только создавать новые таблицы, но не изменять существующие
  (добавить колонку, FK). Alembic генерирует версионированные скрипты миграций
  с `upgrade`/`downgrade` — как git для схемы БД. Используем async-шаблон (`alembic init -t async`).

- **bcrypt напрямую, а не passlib.**
  passlib заброшен (последний релиз — 2022) и несовместим с bcrypt >= 5.0.
  Библиотека bcrypt напрямую проще: `hashpw()` и `checkpw()` — две функции вместо обёртки.

- **PyJWT для JWT-токенов, а не python-jose.**
  python-jose не обновляется (2022, есть CVE). PyJWT минимальный, активно поддерживается.

- **Rate limit: свой in-memory middleware, а не slowapi.**
  Цель — понять, как работает rate limiting изнутри (sliding window log).
  Написан как чистый ASGI-middleware (`__call__`), а не `BaseHTTPMiddleware` —
  два `BaseHTTPMiddleware` в стеке Starlette конфликтуют (dispatch не вызывается).
  Ограничение: не работает с несколькими инстансами (для прода нужен Redis).

- **Эндпоинты: что защищено, что публичное.**
  `POST /links` и `GET /links` — требуют JWT (создание и просмотр своих ссылок).
  `GET /{code}` (редирект) и `GET /links/{code}` (инфо) — публичные,
  иначе короткие ссылки бесполезны.

- **`authenticate_user` возвращает одинаковый ответ при неверном email и пароле.**
  Не даём атакующему перебирать зарегистрированные email.

## 2026-07-03 — Схема таблицы `links`

- **`short_code` — случайная строка (7 симв., base62), а не base62(id).**
  Не палит количество ссылок в сервисе; последовательный ID был бы уязвим к перебору (IDOR).
  Цена — нужна проверка уникальности при вставке и retry на редкую коллизию.
- **`clicks_count` — счётчик-поле в самой таблице `links`, а не отдельная таблица `clicks`.**
  Для MVP достаточно общего числа переходов. Историю по времени (для графиков) добавим
  отдельной таблицей позже, если понадобится — это будет аддитивное изменение, ничего не сломает.
- **`user_id` — колонка есть уже сейчас (nullable, без FK), хотя авторизации ещё нет.**
  Так на этапе 3 не придётся переделывать таблицу `links` — просто добавим FK на `users.id`
  и начнём его заполнять.
- **`domain` — колонка есть уже сейчас, `NOT NULL`, уникальность составная `UNIQUE(domain, short_code)`.**
  Заложено на будущее: сервис может обслуживать несколько доменов/поддоменов
  (напр. `go.faiuk.me` и чужой `qr.another-domain.de`), и один и тот же `short_code`
  может означать разные ссылки на разных доменах.
  `NOT NULL`, а не nullable — в Postgres `NULL` не равен `NULL` в UNIQUE-ограничениях,
  так что nullable-домен сломал бы защиту от дублей `short_code`, пока домен пуст.
  Сама маршрутизация нескольких доменов (Ingress + TLS per-domain через cert-manager) —
  отдельная задача, отложена до этапа 6 (Kubernetes). Сейчас `domain` всегда заполняется
  приложением одним и тем же значением из `BASE_URL`.

## 2026-07-03 — Старт проекта

- **Стек: FastAPI + PostgreSQL + SQLAlchemy (async).**
  FastAPI — async, встроенная OpenAPI-документация, много материалов.
- **PostgreSQL с первого дня, не SQLite.**
  Всё равно впереди Docker/K8s — привыкаем к «настоящей» БД сразу, чтобы не переписывать.
- **Менеджер зависимостей: uv.**
  Быстрый, современный стандарт, проще всего кладётся в Docker-образ. Альтернативы: pip, poetry.
- **Ruff как линтер и форматтер.**
  Один инструмент вместо связки flake8 + isort + black.
- **Отложено осознанно:** Redis, Alembic-миграции, rate-limiting, CI, pre-commit, тесты —
  подключим на соответствующих этапах, чтобы не утонуть на старте.
